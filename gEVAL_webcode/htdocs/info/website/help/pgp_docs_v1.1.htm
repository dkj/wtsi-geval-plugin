Introduction

The PGP Viewer is the visual representation or the front end of a group of databases used to assist in sequencing and finishing genome projects of various vertebrate and non-vertebrate organisms.

GP is an acronym for "pseudo-golden path", a term used to describe the minimum tiling path used in the assembly of the majority of the viewers. The closely associated term is AGP, in which we have AGP viewers for Human and mouse. The primary difference between the PGP and AGP data is the inclusion of partial incomplete sequenced clones in PGP, whereas AGP contains only finished sequence clones. NCBI has a detailed explanation regarding the AGP format here. Thus a PGP viewer will contain more up to date information compared to an AGP viewer.

One common question is what is "What is the difference between the builds here compared to the one seen in the Ensembl Genome Browser?". First the frequency Ensembl creates and releases a build/freeze is low compared to the PGP viewer. For example, the zebrafish build is released once or twice a year in Ensembl, whereas the zebrafish build aims to be released once a month. This allows for a more up to date representation of the genome. However having said that, while Ensembl can be used in improving sequence quality, the PGP Viewer serves primarily a different purpose compared to Ensembl. Ensembl's larger focus is on deep gene-centric analyses, in which PGP viewer does not perform. This is one reason for a quicker turnaround rate in builds. The PGP viewer focuses on the data and analysis results that can help in the improvment of sequence assemblies and resolve regional discrepencies. More examples of the usage to assist in these issues will be described in subsequent sections

The PGP Viewer is developed and maintained by the Genome Reference Informatics Team. The group provides support to the Genome Reference Consortium (GRC), who's primary focus is working to create assemblies that better represent the diversity and provide more robust substrates for genome analysis.

Front Page

  1. List of organisms and their associated viewers. The main and most recent PGP viewer will be on the top of the list, and depending on organism, followed by previous releases, AGP viewers or WGS assembly viewers.
  2. Updates, problems, outages will be listed as news items in this section. This is linked to the PGP Blog and almost all entries will be published on the twitter account. This allows notifications if the PGP viewer happens to be down or of any issues.

Organism Specific Page

The front page of each organism viewer includes a short description of the project and any associated external links. Central to this page is the search bar allowing you to search clone, read, location and other data. The search function will return results with the occurence of the search term, for example the search for zFD282F19 would return not only the clone, but the contigs and the all the other mappings that contain this term, listed and separated by their appropriate categories. The side navigation bar is the entry to the genome browser. The karyotype link returns a dynamic image of the karyotype that can be clicked or a region dragged to enter the browser. The quickest way to begin looking at the data is to click on the location link which jumps to a sample location. There is also a tab with the section as well for navigation.

Karyotype

Click and drag a region to navigate to the specific region. There is also a drop down menu for chromosomes/objects not in the image. This usually pertains to un-localized regions.

Sample Location

This link acts as a quick link to get into the browser. It will jump into a pre-determined region. From there, different analysis tracks can be turned on and off and a different region can be selected.

Navigation Tabs

The left navigation bar is present throughout most of the browser.

Karyotype Page:
returns to the Karyotype region page.
Chr Summary:
Like the Karyotype page, but specifically for the individual chromosome/object currently the browser is on. Includes statistics for the chromosome.
Region Overview:
The broadest view. Here the chromosome can be viewed at no size limits. However because of this only a limited number of tracks are available, and this varies with each organism. Examples include JIRA tickets, fpc contigs and clones.
Region in Detail:
The primary view. Currently a max limit of 2mb size viewable window is allowed. Here is where you can turn on all the tracks via the configure page on the left navigation bar.
PGP Compara:
PGP Compara opens up a view for comparing alignments between assemblies that exist in the viewer.
Marker list:
will return all mapped markers in the region that is being viewed, and links to the appropriate marker information.
Configure this Page:
Use this tab to configure tracks on the browser. You can also use this to configure the look of the browser, such as increasing the viewable size of the browser window.
Export data:
Currently you can use this to dump the sequence in fasta format to your web browser or a text file for the region you are viewing.

Region Overview

The Region Overview allows a global snapshot of the chromosome. Here only a limited number of tracks are available depending on organism. FPC contigs, JIRA tickets and clone information is available. Examples of usage of this page is to quickly glance where a) where all the JIRA issues are b) where all the FPC contig gaps are and c) which clones are finished or unfinished or where clones have been picked for sequencing.

Region in Detail

The primary browser window to view analyses. This consists of two windows, representing different size viewpoints. Both windows are draggable and clickable. Dragging into zoom in or out of the top window will reflect the bottom window. All tracks for both windows are configured via the Configure this page tab on the left navigation bar. Specific navigation using coordinates are also available. The bottom window has as its base the path of clones/contigs and all analysis tracks are of respect to this. The top window, as previously stated is a larger window size, and includes the clone/contigs and its relation to its scaffold/superscaffold level, or in most cases the fpc contigs.

Mapped Ends

One of the most asked question is the colour coding of the mapped ends. The above table lists the colours. Most of the mapped end analyses are separated into a 'rep' and 'no_rep' set. The two sets represent filtered results based on a top limit, followed by times mapped, with no_rep being the filtered set and the rep being all remaining results.

Other Mappings (varies with each organism)

All objects/glyphs are clickable and will return a pop up menu (zmenu) with relavant information and links.

  • cDNA/transcript mappings

    • Green: complete mapping
    Orange: incomplete mapping (coverage etc.)
    zmenu: returns coordinate information for the matching results, coverage, percent identity and links to records of the cdna/transcript and where it may also map in the genome.

  • Self Comparisons

    • Gap Closure

    The selfcomp track shows alignments between clones/contigs against all others in that current assembly. This can reveal overlaps between neighboring clones and haplotypic or troublesome regions. The above example indicates that the two clones are very similar, recipricol alignments to be exact. This is because perhaps one overlaps another, or that the overlap was not caught because of high variation.

    Green - if the self comp hits a clone that is not only a neighbour but 500kb away (this is arbitrary and can be changed). Rational is to capture entire contigs that can be flipped with the right clone being adjacent.
    Orange - hits a clone that is still on the same chromosome but not in the vicinity of 500kb.
    LightBlue - hits a clone in an alternate region (non-chr).
    Red - hit elsewhere.
    zmenu: returns coordinate information for the matching results, and links to the corresponding hit in the genome itself.

  • Markers

    • Magenta: indicates marker map location and current location is the same
    Blue: indicates marker map location and current location is different
    Gray: no marker map location
    zmenu: returns a link to a page for detailed information regarding the marker such as map name, primers and synonyms.

  • Repeats

    • Orange: acrocentromeric repeat
    Teal: centromeric repeat
    Green: telomeric repeat
    Gray: default repeat colour
    zmenu: returns length, type and class of the repeat.

  • Optical Maps

    • The Optical Map data was provided by the Schwartz Lab. Further information regarding the technology can be found here. The optical map tracks can be found in the submenu "Optical Maps" if available. The tracks in this section can be grouped into the categories:

    • insilico digest in which the map was created on
    • specific cell line (eg GM18994) composing of tracks for
      • unaligned fragments
      • aligned fragments

  • Pipeline clone tracks

    • There are four tracks showing the positions of clones in the sequencing pipeline based on end-pair locations.

    • TPF clone: Clones in a current TPF
    • Active non-TPF clone: Clones not in any current TPF that have not been cancelled.
    • Cancelled non-TPF clone: Clones not in any current TPF that have been cancelled, and which do not have an INSDC accession
    • Cancelled +seq non-TPF clone: Clones not in any current TPF that have been cancelled, and which have an INSDC accession.

    "Cancelled" clones are defined as those whose current project status is one of 'Shotgun on hold', 'On hold', or 'Cancelled'.

    For all these tracks, the glyphs are shown in the same way:

    • Shape:
      • If both clone-ends match to sequence within 1 Mb of each other, the glyph is shown as a rectangle stretching between the end-positions.
      • If there is only one clone-end, or the clone-ends lie more than 1 Mb apart, then each clone-end has a glyph with an arbitrary length of 40 kb and an arrow on one end. The square end is at the position of the end, and the arrow shows the direction the clone extends towards from that point.

    • Border colour:
      • Green: The clone is in a current TPF.
      • Yellow: The clone used to be in a TPF, but now is not.
      • Red: The clone has never been in a TPF.
    • Area colour:
      • Grey: The ends are not repetitive.
      • Pink: The ends are repetitive. This is displayed for any clone that has more than a single end or pair of ends.

    zmenu: The menu that comes up when you click on these glyphs is largely self-explanatory.

    • For repetitive cases, links are given to all occurrences of the end(s) associated with this glyph.
    • Where the clone is associated with a JIRA ticket, a link is provided to that ticket. Note that the assignment to JIRA tickets only works if the text of the JIRA ticket explicitly mentions this clone by its internal or external name, or its accession; clones which merely fall into an area spanned by a JIRA ticket will NOT receive a link to that ticket in this menu.

Supplementary Tracks

  • JIRA

    • The JIRA ticketing system is what the GRC uses to track all issues in the reference genome assemblies for human, mouse and zebrafish. Linking out to the official system requires GRC curator access and this is not available to the public. zmenu: returns link to JIRA ticketing system, and the current status of the issue.

  • Clone and Overlaps

    • Navigate Overlap

    Clone track displays the clone with respect to the assembly, this shows the switchpoints used between the neighboring clones and the overlapping region can be viewed via the overlap track. Clones that are in the sequencing pipeline are also present in gapped regions of the assembly. To see a much global picture of perhaps regions that need to be concentrated on, Region Overview can be used with the clone track on to see all relevant information.

    Clones
    Clones are generally gold in colour with the exception to zebrafish where a heatmap colouring system is employed to represent the double haploid fish sequence content in the clones (blue being low and red being high). More information about the zebrafish genome sequencing project can be found here link. Gray coloured clones represent clones that are in the pipeline but have not been sequenced. zmenu: clone information such as alternate names, library, status and prject tracking link.

    Overlaps

    Green
    : Overlap variation < 0.4% variation
    Red
    : Overlap variation > 0.4% variation
    Orange
    : False Gap
    zmenu: returns length,variation and navigation links for the alignment, and to the previous, next good or bad overlap.

  • Insilico Digest

    • Insilico digest using restriction enzyme recognized cut sites.

    • SwaI - Human, Mouse.
    • SpeI - Human.
    • BamHI - Zebrafish.

    Additional enzyme cut sites can be added upon request.

    zmenu: returns fragment size.

PGP Compara

Compara & Patch Example

The PGP Compara page allows comparative analysis between assemblies. Depending on the organism and pgp build in question multiple assemblies can be attached to be analyzed at the same time.

Selection of assemblies that have comparative analysis can be turned on via the Select species button.

In the example above three wgs assemblies are compared. The blocks labelled as REF are the assembly being browsed as reference compared to the the wgs. Tracks can be added or removed via the "configure this page".

Marker List

The table on this page provides a comparison between the physical location and the genetic map location of the aligned markers. The ability to view the genetic map location can help coordinate the potential re-arrangement of sequence components in the genome.

Columns:
  1. Name of the marker and link to its corresponding marker page.
  2. Rank/order of markers in region.
  3. Region start of marker.
  4. Region end of marker.
  5. Clone or component the marker is mapped/placed to.
  6. Individual marker libraries. With each entry following the format: (chromosome)mp:map position. If the chromosome is different from the current in view it will be highlighted red.

Results can be exported as a comma-delimited file (CSV).

Punchlists

Punchlists are automated lists created with the intention that it gives a quick list to navigate to common issues or regions of interest. Usually separated by chromosomes, analysts responsible for specific chromosomes can use this as an opportunity to go through each region of interest one by one.
Current examples include:

Punchlistdescription-
High Variation Overlapslist of all component/component overlaps not meeting variation criteria-
False and True Gapslist of regions where a gap in the assembly is present-
JIRA Ticketslist of Jira tickets, region they are referring to-
Gap Spannerslist of potential gap spanning end reads-
Complete Overlapping/Contained Cloneslist of clones that are in tilepath as contained or being completely overlapped by flanking components-
Selfcomp Overlapslist of selfcomp overlaps on components that potentially align to adjacent clones -

More lists can be added upon request.

Selfcomp Gap Closure Example

Scenario
Clone picked for sequencing to close gap between components may not be necessary
Description
Turning on the selfcomp track reveals green blocks on components adjacent to the gap in question (see image). The green shading of selfcomp indicates matching sequencing in a neighboring region within 500kb. Pressing the result's zmenu will return the hit information indicating the component flanking the gap are of high similarity and possibly can overlap.
Resolution
Re-evaluate the potential for closure and overlap and the actual need/removal for the addition of a clone for sequencing (grey block in clone track)
Tip: Depnding on organism, there may be a selfcomp overlap punchlist which indicates all possible potential overlapping components separated by gaps

PGP Compara & Patch Example

Scenario
PGP Compara can be used to identify regions of difference or variance in alternate assemblies. This can be applied to a fix or novel patch in future reference assembly releases. It can also explain variation within a specific region of interest.
Description

Below are two images, one from GRCH37 and the second the same region but with a GRC Patch (see [add link] regarding patch). The first image shows one clone AC0691.37 and its compara results to the Human Huref and YH1 assemblies. As indicated by the red circles/missing area, there is a region that is not represented by this clone by following the alignment[green shading]

The second image shows the patched region, with an additional clone (AC208170.4) in between the original clone (AC0691.37). As indicated by the green/novel circles, the sequence is now represented in the reference as a patch.
